# Contexto: crawler-machine

## Domínio

Sistema especializado em extrair dados de imobiliárias a partir da URL base de um site.

## Glossário

- **URL base**: endereço inicial do site da imobiliária (ex: `https://imbsmart.com.br`). O ponto de partida para descoberta.
- **Schema do domínio imobiliário**: arquivo global e editável (`config/domain.json`) que lista os campos padrão a serem extraídos de qualquer imobiliária (ex: `quartos`, `valor`, `bairro`, `cidade`) e as configurações globais de LLM, crawler e discovery. Serve de entrada para a geração de schemas específicos por site.
- **Discovery**: etapa de descoberta automática de URLs relevantes a partir da URL base.
- **Schema**: definição estruturada (XPath/CSS) gerada por IA que mapeia onde cada campo de imóvel está no HTML. Apenas as estratégias XPath e CSS produzem schemas persistentes; Fit Markdown e LLM trabalham sobre HTML/markdown já obtido. O artefato `schema.json` passa a conter ambos os schemas sob a chave `schemas.xpath` e `schemas.css`.
- **URLDiscoverer**: classe responsável pela descoberta automática de URLs a partir da URL base.
- **SchemaGenerator**: classe que gera o schema de extração (XPath/CSS) a partir de uma URL de exemplo e dos campos do domínio. Para a cadeia de fallback, gera dois schemas independentes: um XPath e um CSS. O schema usado pelo LLM nativo é derivado dos `FieldConfig` do domínio em formato JSON schema.
- **ImovelCrawler**: classe legada que executa a extração estruturada nas URLs descobertas usando um único schema.
- **CrawlEngine**: classe que orquestra a cadeia de fallback de extração (XPath → CSS → Fit Markdown Regex → Fit Markdown LLM → LLM full HTML) sobre as URLs descobertas. Substitui `ImovelCrawler` como ponto de entrada do crawling nos comandos `run` e `crawl`. Cada etapa pode ser habilitada/desabilitada via `config/domain.json`. Por padrão, XPath, CSS e Fit Markdown Regex estão ligados; Fit Markdown LLM e LLM full HTML estão desligados por consumirem tokens. Apenas a primeira estratégia que faz requisição HTTP possui retry; estratégias subsequentes trabalham sobre HTML/markdown já obtido.
- **DataNormalizer**: classe que normaliza os registros extraídos de acordo com as coerções definidas nos campos do domínio.
- **Exterminator**: componente que filtra registros brutos do crawler antes da normalização, eliminando registros que não contenham os campos obrigatórios mínimos (`bairro`, `cidade`, `valor`, `tipo_imovel`, `url`, `imagem`). Produz a lista de registros sobreviventes e a lista de rejeitados. Atua após a cadeia de fallback completa.
- **Registro rejeitado (rejected record)**: registro bruto eliminado pelo Exterminator por violar as regras de presença mínima. Persistido em `rejected.json` e, no Postgres, mantido como `raw_property` sem correspondente `market_property`.
- **Campos obrigatórios mínimos**: conjunto de campos que um registro bruto deve conter (presentes e não vazios) para prosseguir para a normalização. Todos são tratados igualmente pela cadeia de fallback; não há regras especiais por campo.
- **Pipeline**: orquestrador que executa as etapas em sequência (discovery, schema, crawl, extermination, normalize) e coordena logs/progresso no terminal. A etapa de crawl passa a usar `CrawlEngine` com a cadeia de fallback.
- **CLI**: interface de terminal com subcomandos para executar cada etapa isoladamente (`discover`, `schema`, `crawl`, `normalize`) ou o fluxo completo (`run`).
- **Pasta de saída**: estrutura `output/<slug-do-dominio>/<timestamp>/` contendo os artefatos de cada execução (`discovered.json`, `schema.json`, `raw.json`, `normalized.json`, `rejected.json`, `quality_report.json`, `errors.json`).
- **Fonte**: origem dos dados (ex: nome da imobiliária). Representada por um slug, passado via CLI com `--source-name`.
- **PostgresSink**: destino opcional de persistência. Quando configurado, salva cada execução como um `crawler_run`, os dados brutos em `raw_properties` e os imóveis normalizados em `market_properties`.
- **crawler_run**: registro de uma execução do crawler no Postgres. Apenas o run mais recente com status `completed` de uma fonte está marcado como `latest`.
- **MarketProperty**: imóvel normalizado persistido no Postgres, vinculado a um `crawler_run` e a um `raw_property`.
- **RawProperty**: dado bruto extraído de um imóvel, persistido antes da normalização, permitindo reprocessamento e debug.
- **CatalogRepository**: repositório que carrega os catálogos de normalização (`cities`, `neighborhoods`, `property_types`) em memória para consulta pelos normalizadores semânticos.
- **Fit Markdown**: markdown gerado a partir do HTML bruto da página usando `DefaultMarkdownGenerator` com `PruningContentFilter`, removendo menus, footers e conteúdo de baixa relevância antes das extrações por regex/LLM.
- **FieldNormalizer**: contrato para classes que normalizam um campo individual, retornando `NormalizationResult`.
- **Estratégia de extração**: classe plugável que implementa uma forma de extrair campos de uma página. O sistema prevê cinco estratégias em cadeia de fallback: XPath, CSS, Fit Markdown Regex, Fit Markdown LLM e LLM nativa do Crawl4AI (`LLMExtractionStrategy`).
- **Cadeia de fallback (extraction fallback chain)**: sequência ordenada de estratégias de extração aplicadas a uma mesma página quando campos obrigatórios mínimos não são encontrados pela estratégia anterior. A ordem atual é XPath → CSS → Fit Markdown Regex → Fit Markdown LLM → LLM full HTML. A ativação da última etapa (LLM) é controlada por configuração em `config/domain.json` e pode ser sobrescrita por flag de CLI. Estratégias de fallback aplicam-se apenas aos campos obrigatórios mínimos ainda não encontrados; a engine mantém um mapa de campos pendentes por URL.
- **CrawlResult**: resultado de uma tentativa de extração de uma URL, contendo URL, sucesso, dados extraídos, erro, imagens e, a partir da nova engine, o HTML bruto da página para reaproveitamento pelas estratégias de fallback.
- **NormalizationResult**: resultado da normalização de um campo, contendo valor normalizado, flag de validade, warnings e flag de omissão.
- **ExtractionTrace**: metadado anexado ao registro bruto (`raw.json` e ao `raw_payload` JSONB no Postgres) indicando qual estratégia da cadeia de fallback forneceu cada campo. Removido durante a normalização.
- **ExtractionStrategy**: interface de estratégias de extração. O método principal recebe uma URL e o `CrawlResult` acumulado das estratégias anteriores, devolvendo um novo `CrawlResult` com campos complementares. A engine mantém o registro parcial e passa adiante para preenchimento dos campos ainda vazios.
- **QualityReport**: relatório global de qualidade gerado por execução, com estatísticas de validação por campo, lista de registros com problemas e métricas de uso de cada estratégia da cadeia de fallback, incluindo contador de chamadas LLM.
- **crawler schema**: schema dedicado no Postgres (`crawler`) que abriga catálogos, dados brutos e dados normalizados do crawler, isolados do schema do backend.
- **discovery_runs**: tabela no schema `crawler` que persiste os resultados de descoberta de URLs por fonte. Segue o mesmo padrão `status`/`latest` do `crawler.crawler_runs`, com FK opcional para `crawler.crawler_runs.id`. Permite reutilizar URLs descobertas entre execuções.
- **schema_runs**: tabela no schema `crawler` que persiste schemas de extração gerados por IA por fonte. Segue o mesmo padrão `status`/`latest`, com FK opcional para `crawler.crawler_runs.id`. Armazena `schema_data` (JSONB), `schema_type` (`XPATH`, `CSS`, etc.), `sample_url` e `fields_snapshot`. Na cadeia de fallback, cada schema (XPath e CSS) gera um registro próprio.
- **Prospecting**: módulo de descoberta de imobiliárias candidatas em cidades — uma camada anterior ao pipeline. Dada uma lista manual de cidades (`Cidade,UF`), consulta a Google Places API e produz um YAML de candidatos para revisão humana. Pode persistir os resultados brutos como `Prospect` no banco para evitar redescobrir os mesmos domínios em execuções futuras. A tabela `crawler.prospects` é criada e versionada pelo backend Laravel, não pelo código Python. A estrutura de artefatos em `output/` será reorganizada em pastas por tipo (`runs/`, `prospecting/`, `batch-reports/`) sem manter compatibilidade com caminhos antigos.
- **Place**: resultado bruto do Google Places para uma cidade (`place_id`, nome, `website`, telefone, endereço), atribuído à cidade/UF da query. Quando `website` está ausente, o lugar é descartado na classificação.
- **Candidate**: imobiliária candidata após classificação e filtro. Status `candidate` (aprovado para revisão humana) ou `rejected` com `reject_reason` entre `aggregator`, `no_website` ou `duplicate_domain`. Carrega `base_url` e `source_name` (slug derivado via `build_source_name`).
- **Prospect**: candidato de imobiliária persistido no banco após uma execução de prospecção. Representa tudo que o Google Places retornou para uma cidade, independentemente de ter sido aprovado, rejeitado ou já processado pelo crawler. Serve para evitar reprocessar os mesmos lugares em prospecções futuras. A unicidade é pelo domínio raiz do site (`root_domain(base_url)`), não pelo `google_place_id` — o sistema evita redescobrir o mesmo site, mesmo que o Google Places retorne lugares/filiais diferentes. Se o domínio já consta na base de prospects, ele é excluído de novas queries, sem distinção de status, salvo se o operador usar a flag `--force` para incluir e regravar candidatos já prospectados. Com `--force`, o registro existente é atualizado por upsert real (os dados mais recentes prevalecem). Cada execução de `prospecting find` recebe um `prospecting_run_id` no formato `prospecting_<timestamp>_<sufixo>`, usado para agrupar os prospects daquela rodada. A consulta à base de prospects é feita por um `ProspectRepository`, cuja interface vive no módulo `prospecting` e cuja implementação Postgres fica no pacote `sink`. A interface mínima expõe `filter_new_places` e `save_candidates`. Existem implementações em memória para testes unitários e Postgres para testes de integração. Prospects são persistidos cidade por cidade, após a classificação, incluindo tanto `candidate` quanto `rejected`. Falhas de persistência são tratadas por cidade: o comando continua com as demais e reporta quais cidades não foram salvas. O comando `prospecting find` persiste prospects automaticamente quando o Postgres está configurado; quando não está, executa no modo degradado e gera apenas o YAML.
- **PlacesGateway**: contrato da fonte de descoberta de imobiliárias por cidade. A implementação padrão `GooglePlacesGateway` usa a Places API v1 (`places:searchText`) com `websiteUri` no FieldMask e paginação via `nextPageToken`; o requester HTTP é injetável para testes sem rede.
- **AGGREGATOR_DOMAINS**: blocklist de domínios raiz de portais imobiliários, marketplaces e redes sociais (ZAP, Viva Real, OLX, Facebook, etc.) que não são alvos de crawling. Usada pelo filtro de prospecção para descartar falsos-positivos.

## Decisões arquiteturais

- O crawler continua gerando JSON localmente; o Postgres é um sink opcional ativado pelas variáveis `DB_HOST`, `DB_PORT`, `DB_DATABASE`, `DB_USERNAME` e `DB_PASSWORD`.
- Cada execução com sink cria um `crawler_run` e desmarca o flag `latest` dos runs anteriores da mesma fonte dentro de uma transação atômica.
- Os consumers do backend (AI Searcher e Valuation) leem apenas imóveis do run `completed`+`latest` de cada fonte.
- Discovery e schema generation também são persistidos em tabelas com padrão `status`/`latest`, permitindo reutilização entre execuções. O pipeline reusa resultados anteriores por padrão; flags `--regenerate-discovery` e `--regenerate-schema` forçam nova geração.
- `discovery_runs` e `schema_runs` podem existir independentemente (via comandos `discover` e `schema` com `--save-to-db`) ou vinculados a um `crawler_run` quando gerados dentro do pipeline `run`.
- **Banco de dados é responsabilidade do backend PHP**: criação de schema, tabelas, migrations e seeders do crawler vivem em `ai-backendd-imobiliaria/`. O código Python não deve criar/modificar estruturas de banco em produção; helpers locais em `tests/` existem apenas para viabilizar testes unitários/integrados quando o ambiente Laravel não está ativo.
